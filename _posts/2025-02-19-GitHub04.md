## 1. CD(Continuous Deployment, 지속적 배포) 개요

CD(지속적 배포)는 CI(지속적 통합) 이후 자동으로 코드 변경 사항을 배포하는 프로세스를 의미합니다.

### 1.1. CD의 주요 구성 요소
- **빌드(Build)**: 애플리케이션을 실행 가능한 상태로 패키징
- **테스트(Test)**: 변경 사항의 안정성 확인
- **배포(Deploy)**: 검증된 코드를 프로덕션 환경에 배포

### 1.2. CD의 장점
- **배포 자동화**: 수동 개입 없이 배포 진행 가능
- **빠른 피드백 루프**: 변경 사항이 즉각 반영됨
- **일관된 환경 유지**: 동일한 빌드 및 배포 환경 제공

## 2. GitHub Actions를 활용한 CD 구축

GitHub Actions를 사용하면 코드 변경 시 자동으로 애플리케이션을 배포할 수 있습니다. YAML 파일을 이용해 배포 단계를 정의하고 실행합니다.

### 2.1. 배포 환경 준비
배포 자동화를 위해 필요한 환경 구성:
- **Docker 기반 배포**: 컨테이너 이미지를 빌드하고 배포
- **SSH를 통한 원격 서버 배포**: 서버에 직접 애플리케이션을 배포 및 실행

## 3. GitHub Actions를 활용한 CD 구현

### 3.1. 원격 서버에 직접 배포하는 CD 설정
아래는 `.github/workflows/cd.yml` 파일을 활용하여 원격 서버에 코드를 자동 배포하는 설정입니다.

```yaml
name: Deploy to Server

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v3  # 최신 코드 가져오기
      
      - name: 서버에 배포
        run: |
          ssh -o StrictHostKeyChecking=no ${{ secrets.SSH_USER }}@${{ secrets.SERVER_IP }} "cd /path/to/project && git pull && systemctl restart myapp"
```

### 코드 설명
1. **코드 체크아웃**: `actions/checkout@v3`를 사용하여 저장소의 최신 코드를 가져옵니다.
2. **SSH를 이용한 원격 배포**: SSH를 통해 원격 서버에 접속한 후 `git pull`을 실행하여 최신 코드를 가져오고 `systemctl restart myapp`을 사용하여 애플리케이션을 재시작합니다.


### 3.2. Docker 기반의 CD 설정
Docker를 사용하여 배포하는 경우 아래와 같이 설정할 수 있습니다.

```yaml
name: Docker Deploy

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v3
      
      - name: Docker 이미지 빌드
        run: docker build -t myapp:latest .
      
      - name: Docker Hub 로그인
        run: echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
      
      - name: Docker 이미지 푸시
        run: docker push myapp:latest

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: 원격 서버에서 컨테이너 재시작
        run: ssh ${{ secrets.SSH_USER }}@${{ secrets.SERVER_IP }} "docker pull myapp:latest && docker-compose up -d"
```

### 코드 설명
1. **코드 체크아웃**: `actions/checkout@v3`를 사용하여 코드 가져오기.
2. **Docker 이미지 빌드**: `docker build -t myapp:latest .` 명령어를 통해 최신 코드를 기반으로 이미지를 생성합니다.
3. **Docker Hub 로그인**: GitHub Secrets에서 저장된 Docker Hub 계정을 이용해 로그인합니다.
4. **Docker 이미지 푸시**: 생성된 Docker 이미지를 원격 레지스트리에 업로드합니다.
5. **원격 서버에서 컨테이너 재시작**: 최신 컨테이너 이미지를 가져와 실행합니다.


## 4. 배포 자동화 고급 설정

### 4.1. 배포 실패 시 롤백 기능 추가
배포 실패 시 자동으로 이전 버전으로 되돌리는 기능을 추가할 수 있습니다.

```yaml
  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: 배포 시도
        run: |
          ssh ${{ secrets.SSH_USER }}@${{ secrets.SERVER_IP }} "docker pull myapp:latest && docker-compose up -d || docker-compose up -d myapp_previous"
```

### 코드 설명
1. **배포 시도**: `docker-compose up -d` 명령을 실행하여 최신 컨테이너를 띄웁니다.
2. **배포 실패 시 롤백**: 최신 컨테이너가 실패하면 이전 컨테이너(`myapp_previous`)를 실행합니다.


### 4.2. 배포 알림 설정
배포가 완료되거나 실패할 경우 Slack 또는 이메일 알림을 보낼 수 있습니다.

```yaml
      - name: 배포 성공 알림
        uses: rtCamp/action-slack-notify@v2
        env:
          SLACK_WEBHOOK: ${{ secrets.SLACK_WEBHOOK }}
          SLACK_MESSAGE: "배포가 완료되었습니다."
```

### 코드 설명
1. **Slack 알림 전송**: `rtCamp/action-slack-notify@v2`를 사용하여 배포 완료 알림을 Slack 채널에 보냅니다.
2. **GitHub Secrets 활용**: Slack Webhook URL을 보안 설정을 통해 관리합니다.


## 5. 보안 및 최적화 고려 사항

- **GitHub Secrets 활용**: API 키, SSH 비밀번호 등을 보안 저장소에 보관
- **배포 과정 테스트**: 스테이징 환경을 활용해 배포 검증
- **자동 롤백 설정**: 배포 실패 시 이전 버전으로 되돌리는 기능 추가
- **배포 모니터링**: 로그 분석 및 성능 모니터링 도구 활용

## 6. 결론

GitHub Actions를 활용하면 YAML 설정만으로도 CD(지속적 배포) 프로세스를 자동화할 수 있습니다. Docker 및 원격 배포 방식을 조합하여 효율적인 배포 시스템을 구축할 수 있습니다.
